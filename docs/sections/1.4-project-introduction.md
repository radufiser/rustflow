## Why a Cargo Workspace?

A Cargo workspace lets us manage multiple related crates under a single root. This gives us:

- **Shared dependency versions** — all crates use the same version of `tokio`, `serde`, etc.
- **Single `cargo build` / `cargo test`** — builds and tests everything at once
- **Clear separation of concerns** — the server, database layer, gRPC service, and shared utilities each live in their own crate
- **Faster incremental compilation** — only changed crates are recompiled

We start with two crates and will add more as the course progresses:

```text
rustflow/                       ← workspace root
├── Cargo.toml                  ← workspace manifest
├── crates/
│   ├── rustflow-server/        ← the main HTTP server (binary)
│   │   ├── Cargo.toml
│   │   └── src/
│   │       └── main.rs
│   └── rustflow-common/        ← shared types, utilities (library)
│       ├── Cargo.toml
│       └── src/
│           └── lib.rs
└── README.md
```

## Step 1: Create the Workspace Root

Create a new directory and initialize the workspace `Cargo.toml`. Note that the workspace root itself is **not** a crate — it only defines the workspace members and shared dependency versions.

```bash
mkdir rustflow && cd rustflow
```

```toml name=Cargo.toml
[workspace]
resolver = "2"
members = [
    "crates/rustflow-server",
    "crates/rustflow-common",
]

# Shared dependency versions — all crates inherit from here
[workspace.dependencies]
tokio = { version = "1", features = ["full"] }
serde = { version = "1", features = ["derive"] }
serde_json = "1"
```

### Why `resolver = "2"`?

Cargo's v2 resolver handles feature unification more correctly for workspaces. It avoids a common pitfall where enabling a feature in one crate (e.g., `tokio/macros`) accidentally enables it everywhere. Always use `resolver = "2"` for new workspaces.

### Why `[workspace.dependencies]`?

By declaring dependencies at the workspace level, individual crates can reference them with `dependency.workspace = true` instead of repeating version strings. This ensures consistency and makes version bumps a single-line change.

## Step 2: Create `rustflow-common`

This crate holds shared types, constants, and utilities used across the entire project. We start it nearly empty — it will grow as we add shared error types, configuration structs, and tracing setup in later sections.

```bash
cargo new --lib crates/rustflow-common
```

> **Note:** The `cargo new --lib` command creates a library crate with the proper structure (`Cargo.toml`, `src/lib.rs`) automatically. This is the standard way to create new crates in Rust.

```toml name=crates/rustflow-common/Cargo.toml
[package]
name = "rustflow-common"
version = "0.1.0"
edition = "2021"

[dependencies]
serde = { workspace = true }
```

```rust name=crates/rustflow-common/src/lib.rs
/// RustFlow common library
///
/// Shared types, utilities, and constants used across all RustFlow crates.

/// The name of the application, used in logging, health checks, and metadata.
pub const APP_NAME: &str = "rustflow";

/// The current version of the application, pulled from Cargo at compile time.
pub const APP_VERSION: &str = env!("CARGO_PKG_VERSION");

/// A placeholder health status type. We will expand this significantly
/// in Section 10 when we implement proper health checks.
#[derive(Debug, Clone, serde::Serialize)]
pub struct HealthStatus {
    pub service: &'static str,
    pub version: &'static str,
    pub status: &'static str,
}

impl Default for HealthStatus {
    fn default() -> Self {
        Self {
            service: APP_NAME,
            version: APP_VERSION,
            status: "ok",
        }
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn health_status_defaults_are_correct() {
        let health = HealthStatus::default();
        assert_eq!(health.service, "rustflow");
        assert_eq!(health.status, "ok");
        assert!(!health.version.is_empty());
    }
}
```

### Why start with `HealthStatus` here?

Even though health checks are covered in depth in Section 10, having a basic health response from day one is good practice. It gives us something meaningful to return from our first endpoint and establishes the pattern of putting shared types in `rustflow-common`.

## Step 3: Create `rustflow-server`

This is the main binary crate — the HTTP server that will eventually handle REST, WebSockets, and more. For now, it simply starts up and confirms the workspace is wired together.

```bash
cargo new crates/rustflow-server
```

> **Note:** Without the `--lib` flag, `cargo new` creates a binary crate by default, which is what we need for our server. It automatically generates `Cargo.toml` and `src/main.rs`.

```toml name=crates/rustflow-server/Cargo.toml
[package]
name = "rustflow-server"
version = "0.1.0"
edition = "2021"

[dependencies]
rustflow-common = { path = "../rustflow-common" }
tokio = { workspace = true }
serde = { workspace = true }
serde_json = { workspace = true }
```

```rust name=crates/rustflow-server/src/main.rs
use rustflow_common::{APP_NAME, APP_VERSION, HealthStatus};

#[tokio::main]
async fn main() {
    println!("Starting {} v{}", APP_NAME, APP_VERSION);

    // Verify that shared types from rustflow-common work
    let health = HealthStatus::default();
    println!(
        "Health check: {}",
        serde_json::to_string_pretty(&health).unwrap()
    );

    println!("Workspace is configured correctly!");
    println!();
    println!("Next up: Section 2.1 — we'll turn this into a real HTTP server with Axum.");
}
```

## What We Established

Let's review what we set up and **why**:

| Decision | Rationale |
|---|---|
| Cargo workspace | Manages multiple crates with shared deps; mirrors real enterprise Rust projects |
| `resolver = "2"` | Correct feature unification for workspaces |
| `[workspace.dependencies]` | Single source of truth for dependency versions |
| `rustflow-common` as a library crate | Shared types prevent duplication; enforces a clean dependency graph |
| `rustflow-server` as a binary crate | Clear entry point; depends on common, not the other way around |
| `HealthStatus` in common | Establishes the pattern early; gives us a real type to serialize |
| Tests from the start | Sets the expectation that every crate has tests |

## Dependency Graph

```text

      │
      ▼
rustflow-common
```
