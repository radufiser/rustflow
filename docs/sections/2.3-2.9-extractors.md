# 2.3 - Extractors

## Overview

Extractors are Axum's mechanism for pulling typed data out of incoming HTTP requests. Instead of manually parsing headers, query strings, or JSON bodies, you declare what you need as function arguments — and Axum does the rest.

In this lesson, we'll use three fundamental extractors:

| Extractor | Source | Example |
|---|---|---|
| `Path<T>` | URL path segments | `/tasks/42` → `Path(42)` |
| `Query<T>` | URL query string | `?status=done` → `Query(filter)` |
| `Json<T>` | Request body (JSON) | `{ "title": "..." }` → `Json(task)` |

## Step 1: Define the Task Model

We need task types that will be used across the project. These belong in `rustflow-common` because they'll later be shared with the gRPC service, tests, and other crates.

### Why separate `CreateTask` from `Task`?

This is a critical pattern in API design. The **creation payload** differs from the **stored entity**:

- `id` is server-assigned (auto-incrementing), not client-provided
- `status` defaults to `Pending` on creation
- Later, we'll add `created_at`, `updated_at` — also server-managed

Using the same struct for both leads to awkward `Option` fields everywhere or confusing API contracts.

use axum::{
    extract::{Json, Path, Query},
    http::StatusCode,
    routing::get,
    Router,
};
use rustflow_common::{
    APP_NAME, APP_VERSION, CreateTask, HealthStatus, Priority, Task, TaskFilter, TaskStatus,
};
use tokio::net::TcpListener;

// ── Handlers ────────────────────────────────────────────────

async fn root() -> &'static str {
    "RustFlow is running"
}

async fn health() -> Json<HealthStatus> {
    Json(HealthStatus::default())
}

/// GET /tasks?status=done&priority=high
///
/// The `Query` extractor deserializes the query string into `TaskFilter`.
/// All fields are optional — if no query params are provided, we get
/// an unfiltered request.
async fn list_tasks(Query(filter): Query<TaskFilter>) -> Json<Vec<Task>> {
    // For now, return hardcoded tasks filtered by the query params.
    // In Section 2.5+, this will come from shared state.
    let tasks = sample_tasks();

    let filtered: Vec<Task> = tasks
        .into_iter()
        .filter(|t| {
            filter
                .status
                .as_ref()
                .map_or(true, |s| &t.status == s)
        })
        .filter(|t| {
            filter
                .priority
                .as_ref()
                .map_or(true, |p| &t.priority == p)
        })
        .collect();

    Json(filtered)
}

/// GET /tasks/:id
///
/// The `Path` extractor pulls the `id` segment from the URL.
/// Axum automatically returns 400 Bad Request if the segment
/// can't be parsed as `u64`.
async fn get_task(Path(id): Path<u64>) -> Result<Json<Task>, StatusCode> {
    let tasks = sample_tasks();

    tasks
        .into_iter()
        .find(|t| t.id == id)
        .map(Json)
        .ok_or(StatusCode::NOT_FOUND)
}

/// POST /tasks
///
/// The `Json` extractor deserializes the request body into `CreateTask`.
/// Axum automatically returns 422 if the body can't be deserialized.
async fn create_task(Json(payload): Json<CreateTask>) -> (StatusCode, Json<Task>) {
    // For now, hardcode the ID. In Section 2.6, state will manage this.
    let task = Task {
        id: 999,
        title: payload.title,
        description: payload.description,
        priority: payload.priority,
        status: TaskStatus::Pending,
    };

    (StatusCode::CREATED, Json(task))
}

## How Extractors Work

### The `FromRequest` / `FromRequestParts` Traits

Every extractor implements one of two traits:

```text
FromRequestParts  — can extract without consuming the body
  ├── Path<T>     — from URL segments
  ├── Query<T>    — from query string
  ├── HeaderMap    — all headers
  └── ...

FromRequest       — may consume the request body (can only be used once)
  ├── Json<T>     — parse body as JSON
  ├── String      — body as UTF-8 string
  ├── Bytes       — raw body bytes
  └── ...
```

**Key rule:** A handler can have at most **one** `FromRequest` extractor (because it consumes the body), but **multiple** `FromRequestParts` extractors.

```rust
// ✅ Valid — Path and Query are FromRequestParts, Json is FromRequest
async fn example(
    Path(id): Path<u64>,
    Query(filter): Query<TaskFilter>,
    Json(body): Json<CreateTask>,
) -> StatusCode { ... }

// ❌ Won't compile — two body-consuming extractors
async fn broken(
    Json(a): Json<CreateTask>,
    body: String,  // also consumes body
) -> StatusCode { ... }
```

### Argument Order Matters

Because body-consuming extractors must be **last**, Axum enforces this:

```rust
// ✅ Correct — Json is last
async fn good(Path(id): Path<u64>, Json(body): Json<CreateTask>) -> ...

// ❌ Compile error — Json is not last
async fn bad(Json(body): Json<CreateTask>, Path(id): Path<u64>) -> ...
```

# 2.4 - Request Validation

## Overview

Extractors parse data — but they don't validate business rules. Axum will happily accept `{"title": ""}` as a valid `CreateTask`. We need to enforce rules like:

- Title must be between 1 and 100 characters
- Description, if provided, must be under 500 characters
- Priority must be a valid enum value (already handled by serde)

We'll use the [`validator`](https://crates.io/crates/validator) crate, which provides declarative validation via derive macros.

## Step 4: Create the `ValidatedJson` Extractor

### The Problem We're Solving

Right now, if we wanted to validate a request in a handler, we'd have to do this manually every time:

```rust
async fn create_task(Json(payload): Json<CreateTask>) -> impl IntoResponse {
    // We have to remember to add this in EVERY handler:
    if let Err(errors) = payload.validate() {
        // ... build an error response ...
    }
    // ... actual handler logic ...
}
```

That's repetitive and error-prone — one day you'll forget the check. Instead, let's create a **custom extractor** called `ValidatedJson` that does both steps (parse JSON + validate) automatically.

> **What is an extractor?** In the previous lesson, you used `Json<T>`, `Path<T>`, and `Query<T>` as function arguments. Those are Axum extractors — they automatically pull data out of the HTTP request for you. We're going to build our own extractor that works the same way, but adds validation on top.

### The Big Picture

Here's what we're building and how the pieces fit together:

```text
HTTP Request with JSON body
        │
        ▼
  ┌─────────────────────┐
  │  ValidatedJson<T>   │  ← Our custom extractor
  │                     │
  │  1. Parse JSON ──────────► If JSON is malformed → return 400 error
  │  2. Validate data ──────► If validation fails  → return 422 error
  │  3. Return T ────────────► Handler gets valid data, guaranteed ✓
  └─────────────────────┘
```

### Building It Piece by Piece

**Create** `crates/rustflow-server/src/extractors.rs` — this is a new file. We'll walk through each piece below, but here's the complete file first:

```rust name=crates/rustflow-server/src/extractors.rs
use axum::{
    extract::{rejection::JsonRejection, FromRequest, Request},
    http::StatusCode,
    response::{IntoResponse, Response},
    Json,
};
use serde::de::DeserializeOwned;
use validator::Validate;

/// A JSON extractor that also validates the payload using the `validator` crate.
///
/// Use this instead of `Json<T>` when `T` implements `Validate`.
///
/// # Example
/// ```ignore
/// async fn create_task(ValidatedJson(payload): ValidatedJson<CreateTask>) -> impl IntoResponse {
///     // `payload` is guaranteed to be valid here
/// }
/// ```
pub struct ValidatedJson<T>(pub T);

/// Errors that can occur when extracting and validating JSON.
pub enum ValidatedJsonError {
    /// The JSON itself was malformed or didn't match the expected type.
    JsonError(JsonRejection),
    /// The JSON was valid but failed business rule validation.
    ValidationError(validator::ValidationErrors),
}

impl IntoResponse for ValidatedJsonError {
    fn into_response(self) -> Response {
        match self {
            Self::JsonError(rejection) => rejection.into_response(),
            Self::ValidationError(errors) => {
                let body = serde_json::json!({
                    "error": "Validation failed",
                    "details": errors.field_errors().iter().map(|(field, errs)| {
                        (field.to_string(), errs.iter().filter_map(|e| {
                            e.message.as_ref().map(|m| m.to_string())
                        }).collect::<Vec<_>>())
                    }).collect::<std::collections::HashMap<_, _>>()
                });
                (StatusCode::UNPROCESSABLE_ENTITY, Json(body)).into_response()
            }
        }
    }
}

impl<T, S> FromRequest<S> for ValidatedJson<T>
where
    T: DeserializeOwned + Validate,
    S: Send + Sync,
{
    type Rejection = ValidatedJsonError;

    async fn from_request(req: Request, state: &S) -> Result<Self, Self::Rejection> {
        // First, extract the JSON
        let Json(value) = Json::<T>::from_request(req, state)
            .await
            .map_err(ValidatedJsonError::JsonError)?;

        // Then, validate it
        value
            .validate()
            .map_err(ValidatedJsonError::ValidationError)?;

        Ok(ValidatedJson(value))
    }
}
```

Now let's understand what each part does.

---

### Part 1: The Struct — What `ValidatedJson<T>` Looks Like

```rust
pub struct ValidatedJson<T>(pub T);
```

This is a **tuple struct** — a struct with one unnamed field. Think of it as a simple wrapper around any type `T`. The `<T>` means it's **generic**: `T` can be `CreateTask`, or any other type you want to validate later.

It works exactly like Axum's built-in `Json<T>`, but with validation added. Compare how they're used:

```rust
// Before: Axum's built-in Json — no validation
async fn create_task(Json(payload): Json<CreateTask>) -> ...

// After: Our ValidatedJson — validates automatically!
async fn create_task(ValidatedJson(payload): ValidatedJson<CreateTask>) -> ...
```

The `Json(payload)` / `ValidatedJson(payload)` syntax is called **destructuring** — it unwraps the wrapper and gives you the inner value directly as `payload`.

---

### Part 2: The Error Type — What Can Go Wrong

```rust
pub enum ValidatedJsonError {
    JsonError(JsonRejection),
    ValidationError(validator::ValidationErrors),
}
```

An `enum` in Rust can hold different variants. Our extractor can fail in two different ways:

| Variant | When it happens | Example |
|---|---|---|
| `JsonError` | The JSON itself is broken or doesn't match the struct | `{"title": 42}` (title should be a string) |
| `ValidationError` | The JSON is valid but breaks our rules | `{"title": ""}` (title is too short) |

---

### Part 3: Converting Errors to HTTP Responses — `IntoResponse`

When something goes wrong, Axum needs to know how to turn our error into an HTTP response. That's what the `IntoResponse` trait does.

```rust
impl IntoResponse for ValidatedJsonError {
    fn into_response(self) -> Response {
        match self {
            // Bad JSON? Let Axum's built-in handler deal with it (returns 400)
            Self::JsonError(rejection) => rejection.into_response(),

            // Validation failed? Build a nice 422 response with details
            Self::ValidationError(errors) => {
                // ... builds a JSON body like:
                // { "error": "Validation failed", "details": { "title": ["Title must be..."] } }
                // ... and returns it with status 422
            }
        }
    }
}
```

> **What's a trait?** A trait is like an interface or a contract. The `IntoResponse` trait says: "If you implement me, you promise to provide a `into_response()` method that returns an HTTP response." Axum requires all error types to implement this trait so it knows how to respond to clients.

The `match` keyword is Rust's way of checking which variant an enum holds — similar to a `switch` statement in other languages, but the compiler makes sure you handle every case.

---

### Part 4: Teaching Axum to Use Our Extractor — `FromRequest`

This is the most advanced part. Axum discovers extractors through the `FromRequest` trait. By implementing it, we tell Axum: *"Here's how to create a `ValidatedJson<T>` from an incoming request."*

```rust
impl<T, S> FromRequest<S> for ValidatedJson<T>
where
    T: DeserializeOwned + Validate,  // T must be deserializable AND validatable
    S: Send + Sync,                  // S (server state) must be thread-safe
{
    type Rejection = ValidatedJsonError;  // What error type we return on failure

    async fn from_request(req: Request, state: &S) -> Result<Self, Self::Rejection> {
        // Step 1: Try to parse the JSON body into type T
        let Json(value) = Json::<T>::from_request(req, state)
            .await
            .map_err(ValidatedJsonError::JsonError)?;
        //                                         ^ If JSON parsing fails,
        //                                           wrap the error and return early

        // Step 2: Run validation rules on the parsed value
        value
            .validate()
            .map_err(ValidatedJsonError::ValidationError)?;
        //                                                ^ If validation fails,
        //                                                  wrap the error and return early

        // Step 3: Both passed! Wrap the value and return it
        Ok(ValidatedJson(value))
    }
}
```

Let's break down the new syntax:

| Syntax | What it means |
|---|---|
| `impl<T, S>` | This implementation is generic over types `T` and `S` |
| `where T: DeserializeOwned + Validate` | `T` must implement both `DeserializeOwned` (can be parsed from JSON) and `Validate` (has validation rules) |
| `S: Send + Sync` | The state type must be safe to share between threads (required by Axum) |
| `type Rejection = ValidatedJsonError` | When this extractor fails, it returns our error type |
| `.map_err(...)` | If the `Result` is an `Err`, transform the error into a different type |
| `?` | The question mark operator: if the result is `Err`, return the error immediately; if `Ok`, unwrap the value and continue |

> **The `?` operator in plain English:** Think of `?` as asking "did this work?" — if yes, give me the value; if no, stop here and send back the error. It's Rust's way of avoiding deeply nested `if/else` error handling.

### How It All Fits Together

Here's the journey of a request, step by step:

```text
1. Client sends: POST /tasks  {"title": ""}
                                    │
2. Axum sees the handler wants      │
   ValidatedJson<CreateTask>         │
                                    │
3. Axum calls our from_request()    │
         │                          │
         ▼                          │
4. Step 1: Parse JSON               │
   ✅ JSON is valid syntax    ◄─────┘
   CreateTask { title: "", ... }
         │

5. Step 2: Validate

         │

# 2.5 - Add a Simple Tower Layer (State)

## Overview

Our handlers currently call `sample_tasks()` which returns hardcoded data every time. In a real service, handlers need access to shared data — a database pool, configuration, an in-memory cache, etc. Axum handles this through **state**, which is built on Tower's layer system.

In this lesson, we share a read-only task list across all handlers using `State`.

## What is State in Axum?

State is data you attach to the `Router` that becomes available to all handlers via the `State` extractor. Under the hood, Axum uses a Tower layer to inject the state into every request.

```text
Request arrives
      │
      ▼
┌─────────────┐
│  State Layer │  ← Injects Arc<AppState> into request extensions
├─────────────┤
│   Router     │  ← Matches the route
├─────────────┤
│   Handler    │  ← Extracts State(state) from request extensions
└─────────────┘
```

### Why `Arc`?

Axum clones the state for every request. Without `Arc`, we'd clone the entire `Vec<Task>` per request — expensive and wasteful. `Arc` (Atomic Reference Count) gives us a cheap pointer clone instead.

```text

Handler 2 ──► Arc ─┤──► AppState { tasks: [...] }
Handler 3 ──► Arc ─┘
                    (single allocation, shared across all handlers)
```

# 2.6 - Add a Simple Tower Layer (Mutable State)

## Overview

Read-only state is useful, but most services need to **modify** shared data — add a task, update a status, increment a counter. In async Rust, this requires synchronization primitives because multiple requests can arrive concurrently.

We'll upgrade our state to use `Arc<RwLock<T>>` so handlers can read *and* write.

## Understanding the Concurrency Challenge

```text
Request A (create task) ──────┐
                               ├──► Both try to modify tasks at the same time!
Request B (create task) ──────┘
```

Without synchronization, we'd get data races. Tokio's `RwLock` solves this:

| Lock Type | Behavior |
|---|---|
| `RwLock::read()` | Multiple readers allowed simultaneously |
| `RwLock::write()` | Exclusive access — blocks all readers and writers |

This is ideal for our use case: reads (list/get tasks) are frequent and can happen in parallel, while writes (create/update) are less common and need exclusive access.

### Why `tokio::sync::RwLock` instead of `std::sync::RwLock`?

The standard library's `RwLock` blocks the OS thread while waiting for the lock. In an async context, that would block the Tokio worker thread and stall other tasks. Tokio's `RwLock` yields the async task instead, allowing other futures to make progress.

> **Rule of thumb:** If you hold the lock across an `.await`, use `tokio::sync::RwLock`. If the lock is held only for quick, synchronous operations (no `.await`), `std::sync::RwLock` is actually faster. For our handlers, we'll use Tokio's version to stay safe as the code evolves.

```rust name=crates/rustflow-server/src/main.rs
use axum::{
    extract::{Json, Path, Query, State},
    http::StatusCode,
    routing::get,
    Router,
};
use rustflow_common::{
    APP_NAME, APP_VERSION, CreateTask, HealthStatus, Task, TaskFilter, TaskStatus,
};
use std::sync::Arc;
use tokio::net::TcpListener;

mod state;

    "RustFlow is running"
}

    Json(HealthStatus::default())
}

async fn list_tasks(
    State(state): State<SharedState>,
    Query(filter): Query<TaskFilter>,
) -> Json<Vec<Task>> {
    // Acquire a read lock — multiple readers can execute concurrently
    let state = state.read().await;

    let filtered: Vec<Task> = state
        .tasks
        .iter()
        .filter(|t| filter.status.as_ref().map_or(true, |s| &t.status == s))
        .filter(|t| filter.priority.as_ref().map_or(true, |p| &t.priority == p))
        .cloned()
        .collect();

    Json(filtered)
}

async fn get_task(
    State(state): State<SharedState>,
    Path(id): Path<u64>,
) -> Result<Json<Task>, StatusCode> {
    let state = state.read().await;

    state
        .tasks
        .iter()
        .find(|t| t.id == id)
        .cloned()
        .map(Json)
        .ok_or(StatusCode::NOT_FOUND)
}

async fn create_task(
    State(state): State<SharedState>,
    ValidatedJson(payload): ValidatedJson<CreateTask>,
) -> (StatusCode, Json<Task>) {
    // Acquire a write lock — exclusive access, blocks all other readers/writers
    let mut state = state.write().await;

    let task = Task {
        id: state.next_id,
        title: payload.title,
        description: payload.description,
        priority: payload.priority,
        status: TaskStatus::Pending,
    };

    state.next_id += 1;
    state.tasks.push(task.clone());

    (StatusCode::CREATED, Json(task))
}

async fn delete_task(
    State(state): State<SharedState>,
    Path(id): Path<u64>,
) -> StatusCode {
    let mut state = state.write().await;

    let len_before = state.tasks.len();
    state.tasks.retain(|t| t.id != id);

    if state.tasks.len() < len_before {
        StatusCode::NO_CONTENT
    } else {
        StatusCode::NOT_FOUND
    }
}

    let state: SharedState = Arc::new(RwLock::new(AppState::new()));

    let app = Router::new()
        .route("/", get(root))
        .route("/health", get(health))
        .route("/tasks", get(list_tasks).post(create_task))
        .route("/tasks/{id}", get(get_task).delete(delete_task))
        .with_state(state);

    let addr = "0.0.0.0:3000";
    let listener = TcpListener::bind(addr)
        .await
        .expect("Failed to bind to address");

    println!("{} v{} listening on {}", APP_NAME, APP_VERSION, addr);

    axum::serve(listener, app)
        .await
        .expect("Server error");
}

3. **Think about lock contention** — what happens if `create_task` takes 5 seconds (simulate with `tokio::time::sleep`)? How does it affect `list_tasks`?

# 2.7 - Multiple States - Extension Layers

## Overview

RustFlow doesn't just manage tasks — it also manages **projects** (groupings of tasks) and needs **app configuration** (server name, version, feature flags). Cramming everything into a single `AppState` struct with one `RwLock` creates problems:

- Every handler gets access to everything, even if it only needs config
- Lock contention increases — a write to tasks blocks reads of projects
- Testing becomes harder — you must construct the entire state for every test

Axum supports multiple states through extension layers. Let's split our state into independently lockable components.

## File Changes

## Step 2: Split the State Into Independent Components

**Update** `crates/rustflow-server/src/state.rs` — replace the entire file. Changes from 2.6 are:
- Replaced single `AppState` with `TaskState`, `ProjectState`, and a new composite `AppState`
- Each mutable store has its own `Arc<RwLock<>>` so locks are independent
- `AppConfig` has no lock — it's immutable after startup

```rust name=crates/rustflow-server/src/state.rs
use rustflow_common::{AppConfig, Priority, Project, Task, TaskStatus};
use std::sync::Arc;
use tokio::sync::RwLock;

// ── Task State ──────────────────────────────────────────────

/// State for task management. Independently lockable.
#[derive(Clone)]
pub struct TaskState(pub Arc<RwLock<TaskStore>>);

pub struct TaskStore {
    pub tasks: Vec<Task>,
    pub next_id: u64,
}

impl TaskStore {
    pub fn new() -> Self {
        let tasks = vec![
            Task {
                id: 1,
                title: "Set up workspace".into(),
                description: Some("Initialize the Cargo workspace".into()),
                priority: Priority::High,
                status: TaskStatus::Done,
            },
            Task {
                id: 2,
                title: "Add Axum".into(),
                description: Some("Create the HTTP server".into()),
                priority: Priority::High,
                status: TaskStatus::InProgress,
            },
            Task {
                id: 3,
                title: "Write extractors lesson".into(),
                description: None,
                priority: Priority::Medium,
                status: TaskStatus::Pending,
            },
        ];
        let next_id = tasks.len() as u64 + 1;
        Self { tasks, next_id }
    }
}

impl TaskState {
    pub fn new() -> Self {
        Self(Arc::new(RwLock::new(TaskStore::new())))
    }
}
